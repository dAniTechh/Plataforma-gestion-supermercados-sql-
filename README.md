# Plataforma-gestion-supermercados-sql-

Proyecto de Base de Datos: Sistema de Gestión para Supermercados
Este proyecto consiste en el diseño e implementación de una base de datos relacional orientada a gestionar las operaciones integrales de una cadena de supermercados. El objetivo principal de la práctica ha sido construir una arquitectura escalable y normalizada que abarque todo el ciclo de negocio, desde el abastecimiento y control de inventario hasta las ventas finales, la gestión del personal y la contabilidad corporativa.

El esquema relacional está estructurado en varios módulos funcionales interconectados. Por un lado, cuenta con un catálogo detallado de productos, proveedores y categorías nutricionales. Por otro, integra un sistema de logística que controla el stock físico distribuido entre diferentes almacenes y tiendas, gestionando también las empresas de transporte asignadas. Además, incorpora un módulo completo de recursos humanos para administrar la plantilla, jerarquías de mando, contratos y nóminas, así como un sistema de ventas e-commerce que registra el historial de clientes y pedidos online. Adicionalmente, se han incluido tablas destinadas al control financiero mensual y la gestión de campañas publicitarias.

Para demostrar la viabilidad y robustez del modelo, el script incluye la inserción de datos de prueba y un bloque final con consultas analíticas orientadas al Business Intelligence. Estas sentencias SQL aplican uniones múltiples (JOINs), subconsultas y funciones de agregación para extraer métricas clave del negocio. Entre los reportes generados se incluyen el balance global de ingresos y gastos, la identificación de las categorías de productos más demandadas a nivel nacional, y la distribución de la fuerza laboral y directiva por provincia.

El proyecto está desarrollado íntegramente en SQL y está diseñado para ser compatible con sistemas de gestión de bases de datos relacionales estándar. Para su ejecución, basta con importar y ejecutar el script principal, el cual se encarga automáticamente de levantar la estructura de tablas, establecer las restricciones de integridad referencial, poblar la base de datos y generar los reportes analíticos finales.

A nivel de arquitectura técnica, el modelo de datos hace un uso extensivo de buenas prácticas y reglas de normalización para garantizar la consistencia y escalabilidad del sistema. Para resolver las múltiples relaciones de muchos a muchos (N:M) presentes en la lógica de negocio, se han implementado tablas puente con claves primarias compuestas, como es el caso del control de stock en diferentes ubicaciones o la gestión de líneas de pedido. Además, se ha priorizado la integridad de los datos aplicando restricciones únicas (UNIQUE) en campos de identificación críticos, previniendo la duplicidad de registros en el personal y la clientela. Por otro lado, el manejo de información financiera se ha tipificado estrictamente utilizando formatos precisos (DECIMAL(10, 2)) para evitar los errores de redondeo inherentes a los tipos de coma flotante. Finalmente, la extracción de datos no se limita a uniones estándar, sino que integra subconsultas anidadas de forma dinámica para optimizar el filtrado en las cláusulas de condición.
